
```

```

**构建系统**: CMake + Ninja/Make  
**输出目录** (以 APP debug 为例): `Build/Output_GM_VCUPLUS_APP_TC397X_debug/SoftwarePackets/Hex_Files/`

### LD 文件

链接脚本（.ld 文件）位于：
- `Autosar/Bosch_restricted/Bosch_MCAL/MCAL_CFG/VCUPLUS/startup/CNCONVBASE/GM/`

MAP 文件包含每个 section 的起始地址、大小、以及每个符号的具体地址。

## 2. Flash 分区布局

基于三段链接脚本的分析：

### 整体 Flash 地址空间 (PFlash0, TC39x)

| 分区 | 起始地址 | 结束地址 | 大小 | 说明 |
|------|---------|---------|------|------|
| **BM** | 0xA0080000 | 0xA008FFFF | 64KB | Boot Manager |
| **PDS** | 0xA0300000 | 0xA04B5FFF | ~1.7MB | PDS 区 |
| **KDS** | 0xA0090000 | 0xA0093FFF | 16KB | KDS 数据（板号、序列号等） |
| **BL (FBL)** | 0xA0094000 | 0xA0113FFF | ~512KB | Flash Bootloader |
| **APP** | 0x80114000 (cache) / 0xA0114000 (non-cache) | ~0x80F7FFFF | ~15MB | Application |
| **UCB** | 0xAF400000 | 0xAF405FFF | 24KB | 用户配置块 (BMHD) |
| **DFlash0** | 0xAF000000 | 0xAF0BFFFF | 768KB | Data Flash |
| **HSM** | 0xA0000000 | 0xA007FFFF | 512KB | Hardware Security Module |

> 注：TriCore 架构中，PFlash 有 cached 视图（0x80xxxxxx）和 non-cached 视图（0xA0xxxxxx）。在 .s19/.hex 文件中可能使用 cached 地址，MAP 文件中显示的是链接器使用的实际地址。

## 3. 启动流程

```
[上电/复位]
     │
     ▼
SSW (Startup Software, 芯片内部固化的微代码)
     │ 读取 BMHD (UCB 0xAF400000) 中的启动配置
     │ BMHD 中的起始地址指针决定第一段代码入口
     ▼
BM (Boot Manager, 0xA0080200 入口)
     │ 初始化基本硬件
     │ 校验 KDS 区域
     │ 校验 BL (FBL) 的完整性和签名
     ▼
BL (Flash Bootloader, 0xA0094740 入口)
     │ 初始化通信接口 (CAN/以太网)
     │ 支持固件更新 (OTA)
     │ 校验 APP 的完整性和签名
     ▼
APP (Application)
     │ 主应用程序 (VCU 控制逻辑)
     ▼
正常运行
```

**关键点**：
- 硬件（SSW）只读取 UCB/BMHD，不知道 BM/BL/APP 的概念
- BM→BL→APP 的跳转是纯软件函数指针跳转
- 只有 BM 的 s19 文件包含 UCB 区域数据（BMHD），BL 和 APP 的 s19 不包含

## 4. 固件版本号提取方案

### 核心思路

你的想法**完全正确**。可以通过读取 .map 文件获取版本信息的存放地址，然后从 .hex/.s19 二进制文件中读取对应地址的数据来解析版本号。

### 三个分区的版本信息提取

#### BL (Bootloader)

从 BL 的 MAP 文件中可找到：
```
.FBL_VERSION_ID      a0094340  00000004    (4B)
.FBL_VERSION_STRING  a0094100  00000040    (64B)
.FBL_BUILD_TIME_STAMP a0094344  00000020   (32B)
```

MAP 文件中的符号确认：
```
_rom_bl_version_id     = MEMADDR(RegionBlock_FBL_VERSION_ID)        = 0xA0094340
_rom_bl_version_string = MEMADDR(RegionBlock_FBL_VersionString)    = 0xA0094100
```

BL 版本来源：`Autosar/Prebuilts/Bosch_Prebuilts/BL/BuildBlInfo.h`
```c
#define VERSION_ID (0x230000f4u)
#define VERSION_STRING "rb-vcuplus_main_dev_2026.5.18"
#define BUILD_TIMESTAMP "2026-05-18 13:38:23 +08:00"
```

实际验证（从 hex 读取）：BL Version ID = 0x230000F4，Version String = "rb-vcuplus_main_dev_2026.5.1"

#### APP (Application)

从 APP 的 MAP 文件中可找到：
```
.SW_VERSION_ID       8011449c  00000004    (4B)
.SW_VERSION_STRING   801144a0  00000040    (64B)
```

MAP 文件中的符号：
```
___rom_sw_version_id     = MEMADDR(RegionBlock_APPL_SW_VERSION_ID)     = 0x8011449C
___rom_sw_version_string = MEMADDR(RegionBlock_APPL_SW_VERSION_STRING) = 0x801144A0
```

APP 版本来源：`Version.c` 中的 `version_id` (uint32) 和 `version_string` (char[64]) 变量，通过编译时 `-DVERSION_ID=xxx -DVERSION_STRING=xxx` 传入。

#### BM (Boot Manager)

从 BM 的 MAP 文件中可找到：
```
.BmVersionStringGroup a0080040  00000040    (64B)
```

### 推荐的提取方法

#### 方法一: 解析 .map + .hex/.s19 (离线解析，无依赖)

**步骤 1: 从 .map 文件提取版本地址**

根据 partition 类型，在 .map 文件中搜索特定关键字：

| 分区 | 搜索关键字 | 提取内容 |
|------|-----------|---------|
| APP | `.SW_VERSION_STRING` | 地址 (如 801144a0) 和大小 |
| APP | `.SW_VERSION_ID` | 地址 (如 8011449c) 和大小 |
| BL | `.FBL_VERSION_STRING` | 地址 (如 a0094100) 和大小 |
| BL | `.FBL_VERSION_ID` | 地址 (如 a0094340) 和大小 |
| BM | `.BmVersionStringGroup` | 地址 (如 a0080040) 和大小 |

**步骤 2: 从 .hex/.s19 读取数据**

解析 Intel HEX 或 Motorola S-record 格式，提取指定地址的数据：
- 对于 APP 的 cached 地址 (0x8011xxxx)，在 hex/s19 中直接搜索该地址
- 对于 BL 的 non-cached 地址 (0xA009xxxx)，在 hex/s19 中直接搜索该地址

**步骤 3: 解码版本信息**

- Version ID: 4 字节，小端 uint32（GHS 编译器），如 `f4 00 00 23` → `0x230000F4`
- Version String: ASCII 字符串，最大 64 字节，以 null 结尾
- Version ID 来自 `VERSION_ID` 宏，Version String 来自 `VERSION_STRING` 宏

#### 方法二: 通过 TRACE32 读取 (如果已经连接目标板)

```t32
; 读取 APP 版本字符串
Data.LOAD /NoCODE "<path>/<firmware>.out"
PRINT "APP Version ID: %lx"(s %long DATA.LONG(0x8011449C))
PRINT "APP Version String: %s"(s DATA.STRing(0x801144A0))
```

#### 方法三: 使用 Lauterbach API (cmm 脚本) 从文件解析

```t32
; 从 .out/.elf 直接读取符号
Data.LOAD.Elf "&fileToFlash" /NoCODE
ENTRY appVersionID
  VAR.NEW global uint32 versionID
  var.versionID = s %long DATA.LONG(___rom_sw_version_id)
  PRINT "APP Version ID: %lx"(versionID)
  VAR.NEW global char versionStr[64]
  s %string DATA.STRing(___rom_sw_version_string) 64 versionStr
  PRINT "APP Version String: &versionStr"
```

## 6. 关键技术要点总结

1. **LD 文件**：手写的链接脚本，定义了分区内存布局，是编译前的输入文件
2. **MAP 文件**：链接器自动生成的输出文件，记录了每个符号的最终地址
3. **OUT/HEX/S19 文件**：链接器生成的二进制输出，分别对应 ELF+DWARF 格式、Intel HEX 格式、Motorola S-record 格式
4. **版本信息存储方式**：在 C 代码中使用 `#pragma ghs section rodata = ".XXXX"` 将版本变量放入指定的 section，链接器 ld 文件将该 section 分配到固定的 flash 地址
5. **各分区版本获取**：
   - **BL**: 从 `BuildBlInfo.h` 编译时注入，地址固定为 0xA0094100（字符串）和 0xA0094340（ID）
   - **APP**: 从编译 flag 注入，地址固定为 0x801144A0（字符串）和 0x8011449C（ID）
   - **BM**: 地址固定为 0xA0080040（字符串）
6. **地址映射**：APP 在 MAP 中使用 cached 地址 (0x8011xxxx)，BL/BM 使用 non-cached 地址 (0xA0xxxxxx)，hex/s19 中的地址与 MAP 一致

### 回答你的问题
### Q: LD 文件和 MAP 文件分别是什么？需要手写还是生成的？

- LD 文件：链接脚本，手写的（编译前），定义了内存区域和段的映射关系
- MAP 文件：链接器自动生成的（编译后），记录了每个段/符号的最终地址
